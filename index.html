Principios SOLID

Un poco de historia ...

La programación orientada a objetos supuso la aparición de una nueva forma de diseñar aplicaciones

Habilitó a los desarrolladores la posibilidad de combinar datos con el mismo propósito en una única clase de modo que la funcionalidad común quedaba encapsulada dentro de la aplicación a diferencia de lo que sucede con la programación funcional u otro tipo de paradigmas

Un poco de historia ...

Sin embargo, esto no supuso el fin de aplicaciones con una arquitectura errónea o difíciles de mantener

Con el fin de combatir esto, Robert C. Martin propuso 5 principios con el fin de facilitar a los desarrolladores la creación de aplicaciones fáciles de leer y, lo que es más importante, mantenibles a largo plazo

Un poco de historia ...
##### The **_S_**ingle Responsibility Principle (SRP) _"A class should have one, and only one, reason to change"_
SRP

Una clase tiene que tener una sola razón para cambiar

Para eso necesita tener una sola responsabilidad

Mas reponsabilidades nos generan acoplamientos

SRP

Este principio es el principal y mas importante

Fácil de entender, difícil de poner en práctica

Este principio se puede aplicar no solo a Clases sino a microservicios

SRP

Cómo lograrlo según Uncle Bob Martin ? ...

“Reúne las cosas que cambian por las mismas razones y separa aquellas que cambian por razones diferentes”
SRP

La siguiente clase viola SRP ? ...

							}

... SI!

Tiene mas de una responsabilidad

SRP

##### The *O*pen Closed Principle (OCP) _"You should be able to extend a classes behavior, without modifying it"_
OCP

Deberías ser capaz de extender el comportamiento de una clase, sin modificarla

El segundo principio de SOLID lo formuló Bertrand Meyer en 1988 en su libro Object Oriented Software Construction

Las clases que usas deberían estar abiertas para poder extenderse y cerradas para modificarse

OCP

Robert C. Martin defendió en su blog este principio que a priori puede parecer una paradoja

Es importante tener en cuenta este principio a la hora de desarrollar clases, librerías o frameworks

OCP

Continuamos con la clase Vehiculo ...

						
						class Vehiculo {
							Vehiculo(identificador: String){ }
							getIdentificador() { }

OCP

##### The *L*iskov Substitution Principle (LSP) _"Derived classes must be substitutable for their base classes"_
LSP

Las clases derivadas deben poder sustituirse por sus clases base

Este principio fué acuñado por Barbara Liskov

Significa que los objetos deben poder ser reemplazados por instancias de sus subtipos sin alterar el correcto funcionamiento del sistema

LSP

O lo que es lo mismo: si en un programa utilizamos cierta clase, deberíamos poder usar cualquiera de sus subclases sin interferir en la funcionalidad del programa

Según Robert C. Martin incumplir el Liskov Substitution Principle (LSP) implica violar también el principio de Abierto/Cerrado

LSP

Para cumplir con el principio tendremos que seguir con postulados de Steve Fenton

  • Si la superclase tiene un método que acepta un parámetro con un supertipo, cada subclase debería aceptar como argumento tanto un supertipo como un subtipo
  • Si la superclase devuelve un supertipo las subclases deberían devolver una superclase o una subclase
##### The *I*nterface Segregation Principle _"Make fine grained interfaces that are client specific"_
ISP

Interfaces que sean específicas para un tipo de cliente, es decir, con una finalidad concreta

Es preferible contar con muchas interfaces que definan pocos métodos que tener una interface forzada a implementar muchos métodos a los que no dará uso

##### The *D*ependency Inversion Principle _"Depend on abstractions, not on concretions"_
DIP

Depende de abstracciones, no de clases concretas

Los módulos de alto nivel no deberían depender de módulos de bajo nivel. Ambos deberían depender de abstracciones

DIP

Las abstracciones no deberían depender de los detalles. Los detalles deberían depender de las abstracciones

El objetivo de DIP consiste en reducir las dependencias entre los módulos del código, es decir, alcanzar un bajo acoplamiento de las clases
